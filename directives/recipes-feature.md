# Recipes Feature — Working Directive

This document describes the recipe suggestion system: how it fetches data, what it caches, what its limits are, and what's missing. Read this before touching any recipe-related code.

---

## 1. Overview

Recipe suggestions are generated by calling the Spoonacular API with the names of items currently expiring in the user's pantry. The results are displayed in `RecipeSuggestionsPanel` on the dashboard and recipes tab.

The flow is:
1. `SpoilerDashboard` (client) detects which pantry items are expiring (urgency level amber or red).
2. On mount and whenever the expiring item set changes, it calls the `fetchRecipeSuggestions` server action.
3. The server action calls `getRecipeSuggestions` in `src/lib/recipes.ts`, which checks an in-memory cache and, if stale, fetches from Spoonacular.
4. Results are returned to the dashboard and rendered. A skeleton loading state shows while the fetch is in-flight.

---

## 2. Getting a Spoonacular API Key

This project uses the **RapidAPI** version of Spoonacular, not the direct Spoonacular API. The key format and authentication are different.

1. Go to [rapidapi.com](https://rapidapi.com) and sign up or log in.
2. Search for "Spoonacular Recipe Food Nutrition" and subscribe (free tier available).
3. Find your key under the "Header Parameters" section — it's the `X-RapidAPI-Key` value.
4. Add it to `.env.local`:
   ```
   SPOONACULAR_API_KEY=your_rapidapi_key_here
   ```
5. Restart the dev server. Recipe suggestions will begin working immediately.

**Important:** A direct Spoonacular API key (from spoonacular.com/food-api) will return `401 Unauthorized` — it won't work. You must use the RapidAPI key.

If the key is missing or blank, `getRecipeSuggestions` logs a warning and returns an empty array. The app does not crash — the recipe panel simply stays hidden.

## 3. API Limits (Free Tier)

| Limit | Value |
|-------|-------|
| Requests per day | **150** |
| Points per day | **150** (each request costs points; `findByIngredients` costs 1 point per result) |
| Rate limit | No hard per-second limit on free tier |
| Results per call | Up to 100; we request 6 |

**Cost math:** Each `findByIngredients` call for 6 results costs 6 points. At 150 points/day, that's 25 unique ingredient-set lookups per day before the free tier is exhausted.

The 1-hour in-memory cache is critical for keeping under this limit. If the same set of expiring items triggers multiple page loads within an hour, only the first load hits the API. All subsequent loads read from the cache at zero cost.

**Paid tiers** start at $29/month and include 1,500 points/day. If testing triggers `402` or `402 Payment Required` responses from Spoonacular, the daily quota is exhausted. Wait until midnight UTC for it to reset.

## 4. API Endpoint Used

```
GET https://api.spoonacular.com/recipes/findByIngredients
```

**Parameters sent:**

| Parameter | Value | Purpose |
|-----------|-------|---------|
| `apiKey` | `SPOONACULAR_API_KEY` | Authentication |
| `ingredients` | `item1,+item2,+item3` | Comma-plus-separated expiring item names |
| `number` | `6` | Max recipes to return |
| `ranking` | `1` | Rank by maximizing matched ingredients (vs. minimizing missing ones) |
| `ignorePantry` | `false` | Include common pantry staples in matching |

**Response shape (one item):**

```json
{
  "id": 716429,
  "title": "Pasta with Garlic, Scallions, Cauliflower & Breadcrumbs",
  "image": "https://img.spoonacular.com/recipes/716429-312x231.jpg",
  "usedIngredientCount": 3,
  "missedIngredientCount": 2,
  "usedIngredients": [
    { "id": 11215, "name": "garlic", "amount": 3, "unit": "cloves", "image": "..." }
  ],
  "missedIngredients": [
    { "id": 1001, "name": "butter", "amount": 2, "unit": "tbsp", "image": "..." }
  ],
  "likes": 209
}
```

**What this endpoint does NOT return:**
- Cooking steps / instructions
- `readyInMinutes` (cook time)
- Nutrition information
- Serving size

These fields require a separate call to `GET /recipes/{id}/information`, which costs additional API points. We do not make this call. Instead, we link to the full recipe page on Spoonacular (`sourceUrl`).

## 5. Data Mapping

`findByIngredients` responses are mapped to our `Recipe` type in `src/lib/recipes.ts`:

| Recipe field | Source | Notes |
|---|---|---|
| `id` | `String(raw.id)` | Spoonacular integer ID cast to string |
| `title` | `raw.title` | As returned |
| `imageUrl` | `raw.image` | Direct Spoonacular CDN URL |
| `matchedIngredients` | `raw.usedIngredients[].name` | Ingredients from the user's pantry |
| `allIngredients` | `[...usedIngredients, ...missedIngredients].name` | Full ingredient list for display |
| `steps` | `[]` | Always empty — not returned by this endpoint |
| `cookTime` | `'—'` | Not returned by this endpoint |
| `sourceUrl` | Constructed from title + id | Links to full recipe on Spoonacular |

The `sourceUrl` format: `https://spoonacular.com/recipes/{slugified-title}-{id}`. This is the standard Spoonacular URL pattern — not official API output, but stable enough for linking purposes.

## 6. Caching Strategy

**Cache location:** Module-level `Map` in `src/lib/recipes.ts`.

**Cache key:** Expiring ingredient names, sorted alphabetically and lowercased, joined by comma.
- `['Milk', 'eggs']` and `['eggs', 'milk']` produce the same key: `eggs,milk`
- Prevents duplicate API calls when item order changes

**Cache TTL:** 1 hour (`Date.now() + 3_600_000 ms`). After 1 hour, the next request for the same key triggers a fresh API call.

**Cache lifetime:** The `Map` exists in the Node.js process memory. It is:
- Empty on cold start or after a deploy
- Shared across all requests handled by the same server instance
- Not shared across multiple server instances (horizontally scaled deployments)
- Not persisted to disk or Redis

**Cache behavior when API fails:** On any error (network failure, bad status code, quota exceeded), the function returns `[]` and does not populate the cache. The next request will retry the API call immediately.

## 7. File Map

| File | Role |
|------|------|
| `src/lib/recipes.ts` | Spoonacular fetch + in-memory cache. Server-only. |
| `src/app/recipe-actions.ts` | `'use server'` wrapper. The only entry point for client components. |
| `src/components/spoiler/RecipeSuggestionsPanel.tsx` | Renders recipe cards + skeleton loading state |
| `src/components/spoiler/SpoilerDashboard.tsx` | Calls the server action via `useEffect`, owns `recipes` + `recipesLoading` state |
| `src/types/pantry.ts` | `Recipe` interface — `steps`, `allIngredients`, `sourceUrl` |

**Rule:** Nothing in `src/lib/recipes.ts` or anything that imports `SPOONACULAR_API_KEY` may be imported by a client component. The only allowed client-side entry is `fetchRecipeSuggestions` from `src/app/recipe-actions.ts`.

## 8. Component: `RecipeSuggestionsPanel`

**Props:**

```ts
{
  recipes: Recipe[];
  expiringItems: string[];   // for the "Based on N expiring items" label
  loading?: boolean;         // defaults to false
}
```

**Behavior:**
- `loading: true` → renders 3 skeleton cards with pulse animation. Header shows "Finding recipes…"
- `loading: false, recipes: []` → renders nothing (`return null`)
- `loading: false, recipes: [...]` → renders recipe cards

**Each recipe card:**
- Shows image, title, matched ingredient badges (teal)
- "Cook This" button expands the card
- Expanded: shows full ingredient list with matched items highlighted in teal
- If `steps` is empty (always, with Spoonacular): shows "View full recipe on Spoonacular" link instead
- Cook time badge is hidden when value is `'—'`

---

## 9. Server Action: `fetchRecipeSuggestions`

**Signature:**
```ts
fetchRecipeSuggestions(ingredientNames: string[], count?: number): Promise<Recipe[]>
```

**When it's called:** Inside a `useEffect` in `SpoilerDashboard`, gated on `expiringNamesKey` — a memoized string of sorted expiring item names. Fires on mount and whenever the set of expiring items changes.

**Cancellation:** The effect uses a `cancelled` flag to discard stale results if the component unmounts or the ingredient set changes before the fetch completes.

## 10. Known Limitations

**Steps are never shown.** The `findByIngredients` endpoint does not return cooking instructions. To get them, we'd need to call `GET /recipes/{id}/information` for each recipe — 6 extra API calls per trigger. At 150 points/day free, this would exhaust the quota in 4 triggers. The current solution instead links to the full Spoonacular recipe page.

**Cook time is always `'—'`.** Same reason — cook time (`readyInMinutes`) is only on `/recipes/{id}/information`.

**`sourceUrl` is a best-effort construct.** The URL is built from the recipe title slug + ID. It works for the vast majority of Spoonacular recipes but is not official API output. If a title contains non-ASCII characters or special symbols, the slug may not match the real URL.

**Cache is not shared across server instances.** In a multi-instance deployment (e.g. Vercel Edge, multiple containers), each instance has its own cache. An ingredient set that's cached on one instance will still cause an API call on another.

**No persistent cache across deploys.** Every deploy cold-starts the cache. On Vercel's serverless environment, the cache may also expire between requests (functions spin down when idle). For production, consider replacing the in-memory `Map` with a Redis cache or Vercel KV.

**Free tier is easy to exhaust during testing.** 150 points/day = 25 recipe-fetch triggers. If you're testing heavily, set `SPOONACULAR_API_KEY=` (blank) in `.env.local` to disable the API and use zero points. The app falls back to an empty recipe panel.

**Ingredient matching is Spoonacular's, not ours.** We pass raw item names (e.g. "Greek Yogurt", "Chicken Breast"). Spoonacular's matching handles synonyms and variants better than the previous substring approach, but plurals and brand names may still miss.

## 11. What's Missing

- **Cooking steps.** Would require `/recipes/{id}/information` calls, cached separately per recipe ID. Consider fetching lazily only when the user expands a card, and caching the result so a second expand costs nothing.
- **Cook time.** Same as above — available from `/recipes/{id}/information`.
- **Persistent cache.** In-memory cache doesn't survive deploys or serverless cold starts. Vercel KV or Redis would fix this.
- **Dietary filters.** Spoonacular supports `diet=vegetarian`, `intolerances=gluten`, etc. No UI or preference system exists yet.
- **Recipe save/favorites.** Users can't save recipes they like. Would require a new `saved_recipes` table.
- **Larger result set on recipes tab.** The dashboard shows recipes from a shared state of 6. The recipes tab could request more (e.g. 12) if the user has many expiring items.
